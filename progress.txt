## Codebase Patterns
- Neo4j wrapper: Use `Neo4jConnection` class from `src/knowledge/neo4j.js` — takes `{ uri, user, password }` config, has `healthCheck()`, `session()`, `close()`, `getDriver()` methods
- Neo4j session pattern: `const session = connection.session(); try { await session.run(cypher); } finally { await session.close(); }` — short-lived sessions
- Neo4j integer handling: use `record.get('field')?.toNumber?.()` — neo4j-driver returns Integer objects, not JS numbers
- Env type: exported from cli.ts as `Env` — use in tests to construct env objects for command functions
- ESM guard for CLI: cli.ts must guard `main()` auto-execution with `process.argv[1]?.endsWith('cli.ts')` so vitest imports don't trigger it
- Export functions for testability: prefix CLI helper functions with `export` when they need unit testing
- Test pattern: use `afterAll` to clean up connections; collect connections in an array and close them all
- TelemetryEmitter: `Neo4jConnection` constructor takes an optional `TelemetryEmitter`; for CLI usage, omitting it uses the default emitter
- KnowledgePrivacyAuditor: `new KnowledgePrivacyAuditor({ connection })` — `runAllAudits()` returns `KnowledgeAuditReport` with `.findings[]`, `.has_escalations`; pool optional
- Typecheck: `npx tsc --noEmit` — tests excluded from typecheck via tsconfig `exclude`
- Tests: `node --experimental-vm-modules node_modules/.bin/vitest run` — `fileParallelism: false`
- VaultConnector: constructor takes `{ vaultPath, emitter?, materializeOptions? }` — no `initialize()` method; verify path with `stat()` before constructing
- VaultConnector.scan(): returns `VaultScanResult { files: VaultFile[], errors: string[] }` — each file has `.path`, `.frontmatter`, `.body`, `.lastModified`
- MigrationParser: `new MigrationParser()` — `.parse(frontmatter)` returns `{ valid, template_type, data, errors }` — auto-detects type from frontmatter `type` field or `truth_tier` presence
- Mapper API: all 6 mappers take `{ connection, embedding? }` config, `.migrate(frontmatter, body, sourcePath)` returns `MigrationStats { entities_created, entities_updated, claims_created, embeddings_generated, errors }`
- applySchema: `import { applySchema } from './schema.js'` — takes `(connection, emitter?)`, returns `SchemaApplicationResult { constraintsCreated, indexesCreated, errors }`
- ClaimDecomposer: `new ClaimDecomposer({ connection, embedding?, router? })` — `.decompose(text, opts?)` returns `DecompositionResult` — requires XAI_API_KEY for frontier model
- Mappers use MERGE (idempotent): running migrate twice does NOT duplicate entities, but Claims use CREATE (new IDs each time)
- Test cleanup pattern: use `afterAll` to DELETE test nodes with identifiable prefixes (e.g., `WHERE e.name STARTS WITH 'CliTest'`)
- AQMPipeline: `new AQMPipeline({ connection, router, embedding })` — `.query(question)` returns `AQMPipelineResult` with answer, citations, gaps, timing; handles both "aqm" and "simple" classification
- DualWriteSync: `new DualWriteSync({ connection, vault, pool, embedding? })` — `.syncObsidianToNeo4j(filePath)` returns `SyncResult`; `.startWatching()` uses vault's watcher; `.stopWatching()` stops it; `.runMigration()` creates PG sync tables
- VaultConnector watcher: `.startWatching(handler)` watches .md files, ignores dotfiles; `.stopWatching()` closes watcher; _agent_insights/ must be filtered in handler to prevent sync loops

## Codebase Patterns
- TypeScript project uses ESM (`"type": "module"`) with `moduleResolution: "bundler"` in tsconfig
- Use Zod for runtime schema validation of all data structures (events, frontmatter, etc.)
- Telemetry modules live in `src/telemetry/` — each concern in its own file (types.ts, emitter.ts, rotation.ts)
- Non-blocking pattern: wrap all I/O in try/catch, return null on failure, log to stderr
- File paths use `{YYYY-MM-DD}.jsonl` naming convention (no "events-" prefix)
- Tests use vitest with temp directories (`mkdtemp`) for filesystem isolation
- All exports go through `index.ts` barrel files
- Base telemetry path: `~/vega-telemetry/` with subdirs `events/`, `archive/`
- PostgreSQL `BIGINT` columns return strings in node-pg — always use `Number()` coercion in parse functions
- Use `pg` (node-postgres) with default import: `import pg from "pg"; const { Pool } = pg;` (ESM compat)
- Tier 2 tables use `ON CONFLICT DO UPDATE` for idempotent upserts — conflict keys are the UNIQUE constraints
- Database connection: `VEGA_PG_URL` env var for prod, `VEGA_TEST_PG_URL` for tests (default: `postgres://localhost/vega_test`)
- Tier 2 types in `tier2-types.ts`, migration DDL in `database.ts`, CRUD in `tier2-repository.ts`
- Aggregation jobs in `aggregation.ts` — use `{ emitTelemetry: false }` option in tests to avoid side effects
- vitest `fileParallelism: false` required because test files share PostgreSQL database state
- Anomaly detection needs variance in historical data (stddev > 0) to trigger — uniform values produce stddev=0, skipping detection
- Cost estimation: `estimateCost()` uses per-model rates ($/Mtok); local models like `qwen3:32b` cost $0
- `emitter.readEvents(date)` returns events for one day; for hourly aggregation, filter by timestamp range
- Morning Brief module uses raw pool queries (not Tier2Repository) for analytical cross-agent aggregations
- `beforeEach` with DELETE FROM statements cleans table data between tests while preserving schema
- Raw SQL in metrics_query: avoid hardcoded date literals like `date = '2026-03-02'` — pg serializes JS Date objects using local timezone, causing mismatches. Filter by agent_name only or use parameterized queries
- Self-assessment types in `self-assessment-types.ts`, runner in `self-assessment-runner.ts` — use `{ emitTelemetry: false }` in tests
- Weekly reflections in `weekly-reflection.ts` + `weekly-reflection-types.ts` — generator writes files to `~/vega-telemetry/reflections/{agent_name}/`
- Contradiction detection compares reflection content (what_worked_well/what_didnt_work) against actual Tier 2 quality_daily trends — data-driven reflections won't say "no issues" when anomalies exist, so test for "claims positive + declining trends" instead
- Cross-agent patterns require ≥2 agents; complementary pattern checks success_rate gap (≥0.9 vs <0.7)

## 2026-03-03T16:50 EST - US-402
- **What was implemented**: Aggregation jobs (hourly and daily) — the bridge from Tier 1 JSONL to Tier 2 PostgreSQL
  - `runHourlyAggregation()`: reads JSONL events for a specific hour, groups by agent, computes aggregates (action_count, success/error counts, avg/p95/p99 latency, model_distribution, tokens, cost, performance_score), upserts into `telemetry_agent_hourly`
  - `runDailyAggregation()`: reads JSONL events for a full day, computes cost_daily rollups (per agent+model), quality_daily metrics (success_rate, avg_latency_ms, error_rate with trend), and anomaly detection (>2σ deviation from 7-day rolling average on error_count and latency)
  - Both jobs are idempotent via ON CONFLICT DO UPDATE
  - Both jobs log execution to Tier 1 event stream (event_type: system_event, event_subtype: aggregation_complete)
  - `backfillHourly()` and `backfillDaily()`: iterate over time ranges to re-aggregate historical data
  - Cost estimation via per-model rate table (frontier models: $3/$15 per Mtok in/out, local models: $0)
  - Performance score: weighted composite (0.7 * success_rate + 0.3 * latency_efficiency)
  - Trend computation: compares today vs yesterday, >5% delta = improving/declining
  - 14 tests covering all acceptance criteria

  - `src/telemetry/aggregation.ts` (NEW) — hourly + daily aggregation jobs, backfill, anomaly detection

- **Learnings for future iterations:**
  - Tests sharing PostgreSQL must run sequentially (`fileParallelism: false`) — parallel test files doing `dropTier2Tables` + `runTier2Migration` cause race conditions on `pg_type` unique constraints.
  - Anomaly detection with >2σ needs non-zero stddev in historical data. Uniform historical values (all same) produce stddev=0 and skip detection. Tests must include variance.
